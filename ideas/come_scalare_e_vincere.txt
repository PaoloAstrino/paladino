Ecco lo **stack ottimizzato 2026** per replicare il meglio di Gotham a grande scala, open-source e sostenibile da freelance (ma pronto a crescere).

### Architettura complessiva (Lakehouse + Graph)

```
Dati raw (JSON/CSV ANAC, OpenCUP, ISTAT, ecc.)
    ↓
Lakehouse (Delta Lake su MinIO/S3)          ← storage immutabile + versioning
    ↓ (ETL distribuito)
Apache Spark (PySpark) + Neo4j Spark Connector
    ↓
Entity Resolution Pipeline (Spark + embeddings + LLM)
    ↓
Graph Database (Neo4j Infinigraph o JanusGraph/NebulaGraph)
    ↓
GraphRAG Agent (LangGraph + vLLM/Ollama) + GDS
    ↓
Frontend / API (FastAPI + React Flow)
```

### 1. Storage grezzo e processed (Lakehouse)

**Tecnologia principale**: **Delta Lake** (su **MinIO** per on-premise o S3)

**Perché scala**:
- ACID transactions su oggetti
- Time travel + schema evolution
- Partitioning automatico (per anno, regione, CIG, CUP)
- Z-Ordering per query veloci su CF/P.IVA

**Implementazione**:
```python
# Esempio scrittura partizionata
df.write.format("delta") \
  .mode("append") \
  .partitionBy("anno", "regione") \
  .option("delta.columnMapping.mode", "name") \
  .save("s3a://procurement-raw/anac/")
```

**Volume gestito**: facilmente 10–100+ TB senza problemi.

### 2. ETL e Ingest a grande scala

**Tecnologia**: **Apache Spark 3.5+** (o Databricks se vuoi managed) + **Neo4j Spark Connector** (ufficiale, aggiornato 2026)

**Perché è il pezzo forte**:
- Parallelismo massivo (centinaia di executor)
- Bulk load diretto in Neo4j (migliaia di nodi/relazioni al secondo)
- Trasformazione + Entity Resolution in un unico job

**Implementazione pratica**:
- Usa `neo4j-spark-connector` in modalità `batch` con `node` e `relationship` writer
- Per carichi enormi: `foreachPartition` + transazioni batch di 10k–50k elementi
- Esempio codice base:

```python
from pyspark.sql.functions import *
from neo4j_spark import write_nodes, write_relationships

# Nodi Company
write_nodes(
    df_companies,
    "Company",
    keys=["cf"],                    # merge key
    mode="merge",
    batch_size=20000
)

# Relazioni WINS
write_relationships(
    df_wins,
    "WINS",
    source="Company", target="Tender",
    source_keys=["cf"], target_keys=["cig"],
    relationship_properties=["importo", "data"],
    batch_size=10000
)
```

**Scaling tip**: Inizia con Spark standalone o cluster su 4–8 nodi (64–128 GB RAM ciascuno). Per terabyte usa Kubernetes + Spark Operator.

### 3. Graph Database – Scelta per enorme scala

**Opzione principale consigliata (2026)**: **Neo4j 5.x con Infinigraph**

- Novità 2025: **Infinigraph** → architettura distribuita con **property sharding** (proprietà separate dal grafo topologico)
- Gestisce **100 TB+** in un singolo sistema (transactional + analytical)
- Supporta miliardi/trilioni di relazioni con query millisecondi
- Neo4j GDS (Graph Data Science) scalato orizzontalmente

**Implementazione scaling**:
- Causal Clustering (3+ core + read replicas)
- Property Sharding per nodi pesanti (es. Company con tanti attributi ISTAT)
- Vector index nativo per embeddings (milioni/bilioni di vettori)

**Alternative se superi i 100–200 TB o vuoi più economico**:
- **NebulaGraph** → eccellente per trillions of edges, architettura shared-nothing, nGQL (simile a Cypher)
- **JanusGraph** → su backend **ScyllaDB** (molto più veloce di Cassandra) → vero distributed, centinaia di miliardi di vertici/edges
- **TigerGraph** → se l’analytics (PageRank, Louvain, percorsi) è la priorità numero 1

Per il tuo caso (dati pubblici italiani) → **Neo4j Infinigraph** è il compromesso migliore tra produttività (Cypher, Bloom, GDS) e scala reale.

### 4. Entity Resolution a grande scala (il collo di bottiglia)

**Pipeline ibrida distribuita**:
1. **Blocking** → Spark SQL + bucketing su CF/Provincia/ATECO
2. **Candidate generation** → Similarity (rapidfuzz distributed) + embeddings (SentenceTransformers in Spark UDF o Pandas UDF)
3. **Pairwise matching** → Modello leggero (XGBoost su SparkML) o regole pesate
4. **Clustering** → GraphFrames (Connected Components) o Louvain su grafo temporaneo
5. **Refinement** → Batch su Ollama/vLLM (Qwen2.5-72B o Llama-3.1-70B) solo sui casi ambigui (score 0.6–0.9)

**Scaling**:
- Esegui tutto in Spark (distribuito)
- Per LLM judge: usa **vLLM** o **Ollama in batch mode** su GPU separate (multi-node)

### 5. GraphRAG e Query in linguaggio naturale

- **LangGraph** (LangChain) + Neo4j integration
- Per scale enorme: **vLLM** (invece di Ollama singolo) per servire il modello in parallelo
- Usa **Neo4j vector index** + full-text + Cypher generation con self-correction
